# Decisions: Projects Rework (v5)
Spec: Planning/specs/projects_spec.md
Proposal: Planning/proposals/projects-rework.md
Date: 2026-06-05

- **Projects collapsed to one repeatable "base" type per domain, uniform +2 cap** — replaces 45 bespoke named projects with hand-authored effects. Rejected: keeping per-project bespoke effects (too much content to maintain for a demo; the legibility of "build Docks → more Harbor capacity" wins).
- **Projects fused with the domain-cap system.** `cap = base_cap + Σ active-project contribution`; `base_cap = round(initial Σ level × 1.20)`, frozen at cycle 0. Cap is no longer authored data — `data/domains.json` cap values are ignored. Rejected: keeping cap as a static authored budget (projects then wouldn't matter; the whole point is to give projects a job).
- **20% headroom (`CAP_HEADROOM_MULT = 1.20`), as a named constant.** Chosen so domains start ~83% full and projects matter from cycle 1. Flagged trade-off: that trips the behavior engine's Grow→Steal shift (≥90% cap) after a grow or two; 25–30% is the looser alternative if play feels too tight. Made it one constant so it's feel-tunable without logic changes.
- **Cap contribution scales with project health tier** (intact +2 / damaged +1 / critical 0 / destroyed 0), re-derived each cycle. Rejected: all-or-nothing (+2 while active, 0 when destroyed) — chosen the granular version so sabotage chips at a domain's ceiling progressively, not just on the kill. Consequence: each completed base project is its **own instance** with its own health, so `count` is the number of active instances, not a single shared record.
- **Build unified to 4 work units**, merging v4's `build_time` + `faction_build_actions`. Two add-paths: faction `BuildProject` (own-domain, `d20+level` vs DC 12, +1 on success — free but chancy) and Mayor buy (50 gold + 1 AP = +1 guaranteed, repeatable per AP). Kept the build roll for texture; "4 cycles" is a best-case floor for faction-only building. Design intent: money buys certainty, labor is free but unreliable — two genuinely different paths.
- **`build_cost` (upfront lump) dropped for base projects.** The old "50 gold = −1 cycle" accelerator becomes *the* Mayor build mechanic, generalized to repeat per action point.
- **Initiation: no gate, no cost; NPCs initiate under near-cap pressure** (own domain `utilization ≥ 0.85 × cap`, none under construction). Rejected: the v4 `floor ≥ 3` gate (floor is gone) and Mayor-only initiation (contradicts "factions initiate freely"). At most one base project per domain under construction at a time. Mayor may initiate any domain any time.
- **`tax_collection` deferred, left on the v4 model.** Consequence: the legacy `build_cost`/`build_time`/`faction_build_actions` fields are NOT removed from the `Project` dataclass — base projects simply don't use them, tax_collection still does. Rejected: ripping the fields out now (would break tax_collection, which is wired into Treasury and works).
- **Maintenance kept** at `2 × active_project_count`/cycle. Provides upkeep pressure so stacked base projects aren't free forever — a brake on unbounded cap growth.
- **Dropped:** `faction_level` projects and all bespoke `ProjectEffect`s. `data/projects.json` (45 entries) is dropped and replaced by the ~8 base types.
- **Deferred (Open Question):** count-scaling difficulty — `count` is exposed but nothing hooks onto it yet.
