# Decision: Rating Ceiling Lowered from 10 to 5

**Date:** 2026-04-17
**Status:** Accepted

## Context

The unit-domain rating and faction rating scale was implicitly 1-10, defined by `_UNIT_WEIGHT_TABLE` and `FACTION_CAPACITY` in `scr/engine/formulas.py` (keyed 1 through 10). In practice the top of the range was rarely reached — seeded city data tops out around rating 7 — and the exponential weight progression (level 10 = 512 weight) produced large, unwieldy capacities that skewed faction comparisons at the top end. A 1-5 range gives a tighter, more legible spread where every level is plausibly reachable and the weight table stays in the 0-16 range.

## Decision

### 1. Rating range is now 1-5

`rating` (float) and `floor` (int) for units-in-domains and factions are bounded above by 5. Grow actions that would take rating above 5.0 clamp at 5.0.

### 2. Weight and capacity tables trimmed

Both tables in `scr/engine/formulas.py` drop keys 6-10:

```python
_UNIT_WEIGHT_TABLE: Dict[int, int] = {
    1: 0, 2: 2, 3: 4, 4: 8, 5: 16,
}
FACTION_CAPACITY: Dict[int, int] = {
    1: 0, 2: 2, 3: 4, 4: 8, 5: 16,
}
```

`unit_weight()` and `get_faction_capacity()` already fall back to `0` for unknown keys via `.get(n, 0)`, so any residual call with level 6+ degrades safely rather than crashing.

### 3. Existing data rescaled proportionally

All existing rating values in `scr/data/` and `scr/data/past_cities/Rivers_Point/` are rescaled via:

```
new_rating = 1 + (old_rating - 1) × 4 / 9
```

Mapping: 1→1.00, 4→2.33, 6.45→3.42, 7→3.67, 10→5.00. Endpoints preserved. Applied to `factions[].rating` and `units[].domains[].rating`. Entrench values untouched.

### 4. DB re-seed

Both official city rows in `scr/city_sim.db` are deleted so `seed_official_cities()` re-loads Twin Cities and Rivers Point fresh from the (now-rescaled) JSON on next API startup.

## Rationale

- **Tighter design space.** Five levels is a cleaner mental model and matches the typical spread seen in real data.
- **Weights stay readable.** Max unit weight is 16 (vs. 512), so faction capacities no longer blow up: a 5-rating faction holds 16 members, comparable to current mid-rating factions.
- **Proportional scaling, not truncation.** Rescaling preserves the relative ordering of all existing characters and factions. Star Tribune stays more powerful than Southside Alliance, just on a compressed scale.
- **Trim over clamp in formulas.** Removing level 6-10 entries from the tables (rather than keeping them as unreachable) makes the ceiling explicit in the code — a reader sees the range immediately.

## Consequences

### Behavior changes
- Faction capacities compress: top-end factions go from ~64 members (rating 6+) to 16 members (rating 5).
- Unit weight totals per domain fall roughly 2-4×, so domain utilization relative to `domain.cap` drops. Current `cap` values may feel loose — a separate decision if `domain.cap` should also shrink.

### Known design edge cases (deferred)
Per user direction, these are flagged but not resolved in this change:

1. **Support Faction pooling past the ceiling.** Spec says 3 × L_n supports pool to L_(n+1). Three L5 supports would pool to L6 — which no longer exists. Current spec example (`actions_spec.md`) uses L5→L6; needs revision.
2. **Obscure-at-max.** Spec says a unit must be "at least 1 level above" their action to fully obscure it. At max level 5, that requires L6, which is unreachable — so top-level actions can no longer be fully obscured. Behavior is mechanically consistent; wording in the spec needs softening.
3. **`domain.cap` values.** Unchanged. May want proportional reduction later so domain utilization pressure behaves similarly to before.
