# Proposals

Forward-looking design docs for features that aren't built yet — new subsystems or
cross-cutting mechanics (e.g. a population mechanic), not edits to existing behavior.

One markdown doc per feature. Free-form while incubating: problem, options, rough data-model
and cycle touchpoints, open questions. When you pick one up to build, finalize a **Spec Impact**
section (which specs change, which are new, what Done-when gets added) — that's what `architect`
works from.

Once shipped, move the doc to `../archive/`; the archived proposal is its decision record.

> Trigger: new subsystem / cross-cutting mechanic → proposal first. A simple edit to existing
> behavior skips this and goes straight to `architect`.
