# Polis

Python simulation engine featuring complex autonomous agent behavior, dynamic
faction influence modeling, and emergent narrative generation.

## Stack
- Engine: Python 3.13 (pure Python, no framework)
- API: FastAPI + Uvicorn
- Database: SQLite via SQLAlchemy
- Frontend: Vue 3 (Options API) + Vite

## Commands
- Test: `cd backend && py -m pytest tests/ -q`
- Run/demo (headless): `cd backend && py main.py --cycles 5`
- Run/demo (UI): `cd backend && py -m uvicorn api.server:app --reload` â†’ http://localhost:8000
- Frontend build: `cd frontend && npm run build` (rebuild after `.vue`/`.js` changes, then restart server)
- UI evidence tool: `playwright (python)`   # inspector drives the Vue UI + captures screenshots

## Specs
Source of truth for behavior. Live in `Planning/specs/`, named `feature-name_spec.md`.
Each feature carries inline **Done when:** items tagged `[automated]` or `[human-required]`.

## Reference
Shared as-built truth that specs cite (not testable, no Done-when). Lives in `Planning/reference/`:
`data-models.md` (entities), `formulas.md` (contest math), and `architecture.md` (the system map) â€”
all verified against engine code. Specs reference these rather than redefining them.

## Blueprints
Per-feature build plans generated by `foreman`, in `Planning/blueprints/[feature]_BP.md`.
Format: slices of ordered steps, one concern each, with Build / Test / Done When / Stuck If.
`builder` follows the blueprint; `inspector` checks the spec's Done-when items when done.

## Architecture
The as-built system map: `Planning/reference/architecture.md` (reference-tier, alongside
data-models/formulas). Keep it current when a subsystem is added or the cycle order changes.

## Change rules
Every code, data, or structure change picks a path. History lives in **git** â€” commit each
change with a clear message; there is no separate changelog.

**Quick Path** â€” use only if ALL hold: no files added/removed/renamed, no schema/data-model
change, no core-logic change, nothing a future reader would need explained.
1. Write/edit the code  2. Run the test command  3. Commit
4. If tracked on a blueprint, mark the item done.

**Full Path** â€” everything else (new features, schema, core logic, files added/removed/renamed,
any choice worth recording):
1. Update the spec if one exists â€” if the change reshapes it, run **architect** instead
2. Write/edit the code  3. Run the test command  4. Run **inspector** if the change is tracked
by a blueprint  5. Update this CLAUDE.md file map if files moved; update
`Planning/reference/architecture.md` if a subsystem was added or the cycle order changed
6. Write a decision doc to `Planning/decisions/` if a non-obvious choice was made (what + why)
7. Commit  8. If tracked on a blueprint, mark the item done.

Decision docs: `Planning/decisions/` (`YYYY-MM-DD-title.md`).  Skill output: `output/`.
