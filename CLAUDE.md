# Polis

Python simulation engine featuring complex autonomous agent behavior, dynamic
faction influence modeling, and emergent narrative generation.

## Stack
- Engine: Python 3.13 (pure Python, no framework)
- API: FastAPI + Uvicorn
- Database: SQLite via SQLAlchemy
- Frontend: Vue 3 (Options API) + Vite

## Commands
- Test: `cd backend && py -m pytest tests/ -q`
- Run/demo (headless): `cd backend && py main.py --cycles 5`
- Run/demo (UI): `cd backend && py -m uvicorn api.server:app --reload` ‚Üí http://localhost:8000
- Frontend build: `cd frontend && npm run build` (rebuild after `.vue`/`.js` changes, then restart server)
- UI evidence tool: `playwright (python)`   # inspector drives the Vue UI + captures screenshots

## Workspaces
- `/Planning` ‚Äî Specs, reference, decisions, backlog ‚Äî see `Planning/CLAUDE.md`
- `/backend` ‚Äî Application code (engine, API, db, tests) ‚Äî see `backend/CLAUDE.md`
- `/docs` ‚Äî Long-term goals, architecture (as-built docs)
- `/tools` ‚Äî Audit, validation scripts (`py tools/audit.py` for engine snapshot)
- `/frontend` ‚Äî Vue 3 + Vite browser UI

## Specs
Source of truth for behavior. Live in `Planning/specs/`, named `feature-name_spec.md`.
Each feature carries inline **Done when:** items tagged `[automated]` or `[human-required]`.

## Reference
Shared truth that specs cite (not testable, no Done-when). Lives in `Planning/reference/`:
`data-models.md` (entities) and `formulas.md` (contest math) ‚Äî both verified against engine code.
Specs reference these rather than redefining them.

## Blueprints
Per-feature build plans generated by `foreman`, in `Planning/blueprints/[feature]_BP.md`.
Format: slices of ordered steps, one concern each, with Build / Test / Done When / Stuck If.
`builder` follows the blueprint; `inspector` checks the spec's Done-when items when done.

## Architecture
The as-built system map (documentation, not planning): `docs/architecture.md`.

## Change rules
Every code, data, or structure change picks a path. History lives in **git** ‚Äî commit each
change with a clear message; there is no separate changelog.

**Quick Path** ‚Äî use only if ALL hold: no files added/removed/renamed, no schema/data-model
change, no core-logic change, nothing a future reader would need explained.
1. Write/edit the code  2. Run the test command  3. Commit
4. If tracked on a blueprint, mark the item done.

**Full Path** ‚Äî everything else (new features, schema, core logic, files added/removed/renamed,
any choice worth recording):
1. Update the spec if one exists ‚Äî if the change reshapes it, run **architect** instead
2. Write/edit the code  3. Run the test command  4. Run **inspector** if the change is tracked
by a blueprint  5. Update this CLAUDE.md file map if files moved
6. Write a decision doc to `Planning/decisions/` if a non-obvious choice was made (what + why)
7. Commit  8. If tracked on a blueprint, mark the item done.

Decision docs: `Planning/decisions/` (`YYYY-MM-DD-title.md`).  Skill output: `output/`.

## Skills
Workflow (Plumbline): `scaffold` ‚Üí `architect` ‚Üí `foreman` ‚Üí `builder` ‚Üí `inspector`,
with `surveyor` (spec-vs-code drift) and `walkthrough` (maintenance pass) run when useful.

| Skill | Use when |
|-------|----------|
| `architect` | Define/redefine a feature ‚Üí writes the spec |
| `foreman` | Plan how to build a spec'd feature ‚Üí blueprint |
| `builder` | Write code for one blueprint slice |
| `inspector` | Prove a slice/feature meets its Done-when items |
| `surveyor` | Static spec-vs-code drift report (replaces old `speccheck`) |
| `walkthrough` | Autonomous maintenance pass (replaces old `improve`) |
| `postrunlog` | After a sim run ‚Äî summarize cycle data to `backend/logs` (domain-specific; kept) |

## Naming conventions
- Specs: `feature-name_spec.md`
